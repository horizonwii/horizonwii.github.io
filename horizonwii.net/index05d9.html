 Guide taken from RVLution Wiki and simplified by me

Riivolution does its magic according to XML files found in the folder /riivolution on an SD card or USB drive. This post will attempt to explain you how to write powerful XML files.

If you're just interested in getting Riivolution running, or want to see an example, have a look at this thread. This is meant for power users and people planning on releasing packaged game mods.

The Basics
Riivolution patch XMLs are made up of three parts:

- The id tag, which specifies which game it should be applied to, if any.
- The options tag, which organizes patches into an Option-Choice menu system.
- The patches themselves, which define what files and memory to replace and how.

Parameters
Riivolution provides a few built-in parameters that you can use inside of your XMLs. They describe the disc currently in the drive and are as follows:

- {$__gameid} contains the first three characters of the Game ID. For example, "SMN" for New Super Mario Bros. Wii
- {$__region} contains the region character of the Game ID, usually "E", "P", "J" or "K"
- {$__maker} contains the maker code of the Game ID. For example, "01" is Nintendo.

Using these together can be used to get the full Game ID, "SMNP01" :

{$__gameid}{$__region}{$__maker}

Root
All Riivolution XMLs are encompassed in the wiidisc node, so in this way :

<wiidisc version="1" root="/nsmb" shiftfiles="true" log="true"> ... </wiidisc>

The root node has a few parameters that can be applied to it :

- Version (Required) : Usually you will use version 1, there's not much else to say
- Root (Optional) : This path will be used as the root for all the paths used within this XML. It can be useful to avoid writing long paths over and over. Make sure to put "/" before the folder name.
- Shiftfiles (Optional) : Set to true or false. This is REQUIRED if you are injecting new files into the game, as it will allow Riivolution to move them in.
- Log (Optional) : Enables logging of system events. It's only needed for debugging.

ID
The id tag is meant to make sure an XML is only applied to the game it was written for. If it is ommitted, the XML will be loaded for any game. There can't be more than one ID tag in one XML.

This tag can specify the game ID, developer, disc number, version, and region(s). Any combination of these are allowed, and anything omitted or empty will allow everything through. Here's an example:

<id game="SMN" developer="01" disc="0" version="0"> <region type="P" /> <region type="E" /> </id>

The previous example will make sure the XML will load for both PAL and NTSC-U versions of New Super Mario Bros. Wii. Every parameter here is optional :

- Game : Insert the three characters (or four if you're too lazy to use the region nodes and want to restrict it to one region) to match the disc's Game ID. Usually starts with a 'R' or 'S'.
- Developer : The two numbers at the end of the Game ID used to identify the makers of the game.
- Disc : This specifies the disc number of the game. This is usually 0, but may be 1 for the second disc of a multi-disc game (like Tales of Symphonia)
- Version : This one, instead, specifies the disc version. This is again usually 0, but may be increased for game reprints or remakes.

Options
All sections of an XML must be encompassed by the options tag :

<options> ... </options>

There are no parameters that can be used here.

Sections
A section represents a page in the GUI, though they can be split up into multiple pages if they contain too many options. They contain options, and have no parameters except the name :

<section name="Levels"> ... </section>

Options
An option represents something modifiable in the program. It contains choices, and only one of them can be active at any given time.

<option id="levelpack" name="Level Pack" default="1"> ... </option>

Option tags have three parameters :

- ID (Optional) : This is used for combining options from two different XMLs (they must be under the same section name as well)
- Name (Required) : The name of the option to be shown in the program
- Default (Optional) : This allows to set the default choice to be selected when the program first starts. Once the user has saved their settings (launched a game), this is ignored and the last-used choice is set.

Choices
A choice represents one choice within an option. It also references patches to be applied.

<choice name="Enabled"> <patch id="lvlpack" /> <patch id="asmfixes" /> </choice>

All the parameters are required for the choice to work :

- Name : This is the name of the choice as it appears in the program
- Patch ID : The ID of the patch/es that this choice enables.

Patches
A patch is a set of commands that Riivolution will execute when the corresponding choice is selected. It can contain file, folder, and memory patches.

<patch id="lvlpack" root="/Stage"> ... </patch>

- ID (Required) : This is the id that a choice will reference to in order to enable the patch
- Root (Optional) : This sets the root path for all items in this patch. It can be used as an alternative to the wiidisc root path. As in the other one, do not forget to put "/" before the folder name.

File Patch
A file patch will replace the contents of one file on disc with one from an external device.

<file disc="01-01.arc" external="W1-1.arc" resize="true" create="false" offset="0x10" length="0x100" />

There are several parameters available :

- Disc (Required) : The file on disc to replace. This can be set as a full path or just a filename to search for. Make sure to put "/" before paths, and remember if you set any root path.
- External (Required) : The file on SD/USB to replace the disc file with. The setting works like the previous one.
- Resize (Optional) : Enabled by default. Can be true or false, and it specifies whether to resize the disc file to the external file. This should always be enabled to prevent file corruption.
- Create (Optional) : Disabled by default. Can be true or false, and it specifies whether to add the file on the disc if it does not exist. This is required if the files are new.
- Offset (Optional) : This sets from which part of the file (written in hex) the replacement happens. It's set to the beginning of the file by default.
- Length (Optional) : The length specifies until which point (in hex) to patch the file. It's set to the size of the external file by default.

Folder Patch
A folder patch replaces all the files in a disc folder with the ones in a device folder.

<folder disc="/Stage" external="/Levels" resize="true" create="true" recursive="true" length="0xABCD" />

There are several parameters available :

- Disc (Required) : The folder on disc to replace. Make sure to put "/" before the path and remember if you set any root path.
- External (Required) : The folder on SD/USB to replace the disc folder with. The setting works like the previous one.
- Resize (Optional) : Enabled by default. Can be true or false, and it specifies whether to resize the disc files to the ones found in the external folder. This should always be enabled to prevent file corruption.
- Create (Optional) : Disabled by default. Can be true or false, and it specifies whether to add any files and folders (if Recursive is set to true) to the disc folder if they're not found. This is required if the files/folders are new.
- Recursive (Optional) : If set to true, it tells Riivolution to go looking in subfolders for other files to replace. This only works when the wiidisc or patch nodes have a root parameter appended to them.
- Length (Optional) : The length specifies until which point (in hex) to patch all the files in the folder. It's set to the size of each external file by default.

Savegame Patch
A savegame patch will redirect a game save to the external path specified. It can be used to store a save on SD/USB/Wifi without corrupting your original save when playing an hack with different stages.

<savegame external="/save" clone="true" />

There are two parameters available :

- External (Required) : The folder on SD/USB to store the game's savefile inside. It will be created if it does not exist.
- Clone (Optional) : Enabled by default, specifies whether to copy the Wii's current savegame to the external folder before starting the game. When set to true, this is only done when the external folder is empty, so only on first startup or if it's been deleted. When set to false, it begins an entirely new save if the external folder is empty.

Memory Patches
There are a few different kinds of memory patches, and they all patch the Wii's memory right before a game loads. They can be used to apply various different ASM hacks to games.

The default memory patch looks like the following, and is used to write values to a set address.

<memory offset="0x70ADD5" value="B000B5" original="BABE55" /> <memory offset="1E75A600" valuefile="/ASM.bin" />

There are several parameters available, and you must choose between value and valuefile for each memory patch :

- Offset (Required) : The place in memory at which to write. It must be written in hex and preceded by "0x".
- Value (Required) : The value to write to memory. It must be written in hex, and its length must be a multiple of two.
- Valuefile (Required) : An alternative to value is to specify a file path to read the value from. The file will be read entirely. Don't forget to put "/" before the path and remember whether you've set root paths in the XML.
- Original (Optional) : This specifies the original value used by the game in the selected offset. If it does not match the memory, the value won't be written to it.

Ocarina Patch
An Ocarina Patch will look in the dol file for the specified value, then find the next occurring blr and replace it with a branch to offset.

<memory offset="0xCAD111" value="62EEED" ocarina="true" />

All the parameters are required to make this work :

- Offset (Required) : The offset in memory of the location to branch to. It must be written in hex and preceded by "0x".
- Value (Required) : The valuepattern to search the game binary for.
- Ocarina (Required) : Must be set to true for this form to work.

Search Patch
A memory search patch can be used to replace a pattern in the main executable dynamically.

<memory original="1337BAAD" value="FC0FFB20" align="4" search="true" />

There are several parameters available, and you must choose between value and valuefile for each memory search patch :

- Original (Optional) : The original pattern to search the game binary for.
- Value (Required) : The value to replace the original pattern with. It must have the same length.
- Valuefile (Required) : An alternative to value is to specify a file path to read the value from. The file will be read entirely. Don't forget to put "/" before the path and remember whether you've set root paths in the XML. The length must be the same in this case, too.
- Align (Optional) : The byte alignment to search with. The default is 1, but it should often be higher in order to save time when searching.
- Search (Required) : Must be set to true for this form to work.

That's it ! Hope it helps.

 This tutorial will teach you how to scan your XML for errors in writing.

Requirements :

- An XML file you want to check
- Notepad++
- XMLTools Plugin
- Riivolution Schema

Note : The <memory> tag has no checks for the presence of all the required parameters, so use the information in this thread to check it yourself.

Steps :

- After installing the XMLTools plugin, open the XML you want to scan in Notepad++
- Click on Plugins > XML Tools > Validate Now
- A new window will open
- Browse to the Riivolution Schema XSD
- If everything is correct, a popup will appear telling you the XML is valid, otherwise a window will appear listing the errors you've made

That's it !
